Mackes Audio Platform

An educational Linux audio platform for musicians and audio developers.

MAP2 is built to cover a very wide range of real-world audio jobs without feeling like two different products bolted together.

For a single guitarist, it can be as simple as a small form factor x86_64 PC running as a headless audio processor. Drop it into a pedalboard rig or small rack, hook up an interface, and control everything from a tablet. No screen on stage, no bulky computer setup, just a compact, reliable box handling amp tones, effects, routing, MIDI control, and presets.

At the other end of the scale, the same MAP2 platform can be used as a serious processor for education, installed sound, and stadium-level audio systems. On larger x86_64 hardware, it has the power for much more demanding work: failover, deep MIDI integration, API-based control, AVB support, and complex signal routing across bigger systems.

That range is really the point. MAP2 can start as a personal performance rig for one player and scale all the way up to a powerful audio platform for large, mission-critical environments.

Install MAP on Fedora Server

Start with Fedora Server.

Download Fedora Server

MAP2 is positioned around a Fedora Server base because Fedora Server offers a strong open-source Linux foundation, current packages, real-time-friendly workflows, and a clean headless/server path for audio experimentation.

  1. Download Fedora Server: get the current official Fedora Server image from the Fedora Project.
  2. Install Fedora Server: perform a clean server install on the target machine and apply system updates.
  3. Prepare the audio host: configure networking, storage, and the hardware interfaces you plan to use.
  4. Install MAP2: use the MAP2 nightly release or source repository to continue platform setup and evaluation.

Current framing

MAP2 is strongest as an educational and experimental platform. The codebase shows serious depth and real implementation work, but the current evaluation positions it as best suited for learning, tinkering, prototyping, and expert-led exploration rather than broad production deployment.

Full platform evaluation, in plain language.

MAP2 is not a shallow concept project. It is a technically serious platform with real DSP, plugin, network-audio, MIDI, and deployment breadth. Its main story is not “look at an idea.” It is “look at how a modern Linux audio platform is actually assembled, where it is strong, and where it still needs closure.”

What it already proves

Serious implementation depth

  • Real-time DSP engine: JUCE-based audio processing with low-latency engineering work.
  • Broad control plane: FastAPI routes, WebSocket updates, presets, diagnostics, and remote control.
  • Multiple operating surfaces: browser UI, TUI/SSH workflows, and hardware-oriented control patterns.

Current maturity

Ambitious, useful, still experimental

Core audio and performance

  1. Ultra-Low Latency Audio Engine: JUCE 8.0-based core delivers sub-3ms round-trip latency on optimized hardware.
  2. Real-time SCHED_FIFO Processing: Audio threads run at the highest real-time priority for maximum stability.
  3. CPU Core Isolation: Dedicates specific CPU cores exclusively to audio processing, preventing OS interference.
  4. Memory Locking: Prevents page faults on the real-time audio thread, eliminating potential glitches.
  5. PipeWire-Native with JACK Compatibility: Modern, flexible audio backend for seamless Linux integration.
  6. High-Resolution Audio Support: Process audio at professional sample rates up to 192kHz.
  7. Configurable Buffer Sizes: Tune latency versus stability with buffer sizes from 64 to 256 samples and beyond.
  8. Automatic Plugin Delay Compensation: Maintains perfect phase alignment across the entire plugin chain.
  9. Zero-Allocation Real-time Audio Path: Pre-allocates all memory to prevent dynamic allocations that could cause xruns.
  10. XRun Detection: Actively monitors for audio dropouts and provides detailed logging and recovery.
  11. Professional Metering Suite: Includes Spectrum Analyzer, LUFS Loudness, VU Meters, and Phase Correlation.
  12. Hot-Swappable USB Audio Interfaces: Add or remove audio hardware without restarting the service.
  13. Optimized Eigen-based NAM Inference: Neural Amp Modeler backend is optimized for high-performance CPU inference.
  14. Graceful Hardware Degradation: The system remains operational and controllable even if audio hardware is disconnected.
  15. Fixed-Point and Floating-Point Processing: Supports various bit depths for maximum audio quality.
  16. IRQ Balancing Management: Disables IRQ balancing on dedicated audio cores to ensure consistent low latency.
  17. Asynchronous Audio Device Handling: Audio hardware changes are handled in the background without blocking the main application.

A human-understandable description of the platform.

Mackes Audio Platform is easiest to understand if you think of it as a learning laboratory for modern digital audio. It turns a Linux machine into a configurable audio processor, but the important part is that the system stays open enough to study.

Plain-language description

MAP2 shows how professional-style audio processing can be built from normal computing hardware and open software. Instead of hiding the signal path, scheduler decisions, plugin loading, network interfaces, and control logic, it exposes them. That makes it useful for musicians who want to learn how their tools work, and for audio developers who want to inspect a complete, multi-layer system rather than isolated code samples.

  • For musicians: it behaves like a configurable effects and routing platform that can be explored, not just used.
  • For audio developers: it is a case study in combining DSP, APIs, UI, and Linux system tuning.
  • For students: it demonstrates concepts like real-time threads, plugin hosting, network audio, and control surfaces in one project.
Educational purpose: MAP2 is valuable because it is transparent. It gives learners a working reference for how low-latency Linux audio systems are assembled, where the engineering tradeoffs are, and how audio software, operating systems, and interfaces meet in practice.

Detailed architecture.

The educational value is strongest when the platform is read as three connected layers. Each layer is understandable on its own, but the real lesson is how they interact.

Layer 1

JUCE / C++ audio engine

This is the time-critical part of the platform. It handles DSP, plugin processing, real-time audio callback behavior, signal chains, and low-latency execution concerns.

  • Why it exists: real-time audio work demands predictable execution and tight memory control.
  • What it teaches: callback design, plugin hosting, DSP composition, and performance tradeoffs.
Layer 2

Python / FastAPI control plane

This layer manages the platform. It exposes API routes, handles orchestration, tracks presets and state, coordinates services, and provides a bridge between the engine and the user-facing tools.

  • Why it exists: higher-level control tasks do not belong inside the hard real-time layer.
  • What it teaches: API design, async service boundaries, persistence, and remote control patterns.
Layer 3

Web UI and operator surfaces

The browser interface turns system capabilities into something visible and interactive. MAP2 also extends that idea through TUI and hardware-oriented controls for remote or embedded-style use.

  • Why it exists: a platform this broad needs usable control surfaces, not only config files.
  • What it teaches: real-time UI feedback, operational information architecture, and remote administration.
Cross-layer lessons

What the whole platform demonstrates

  • Separation of concerns: the DSP engine does the time-critical work while Python handles management and the web UI handles interaction.
  • Linux as an audio host: system tuning, service orchestration, and device/runtime behavior matter as much as code.
  • Audio networking: AVB and clustering show how single-box processing can evolve toward distributed workflows.
  • Product honesty: the platform is broad enough to be useful, and unfinished enough to be educational in a very practical way.

Use cases.

MAP2 is most compelling when used as a platform for exploration, learning, and small-scale practical workflows. These use cases align with what the current codebase and documentation show most clearly.

Learning real-time Linux audio

Study how scheduling, isolated CPU work, plugin graphs, and callback timing affect a working DSP platform.

Building guitar and bass rigs

Experiment with amp modeling, impulse responses, effects chains, MIDI control, and preset recall on Linux.

Teaching DSP and systems design

Use one codebase to explain audio processing, APIs, UI control, persistence, and deployment workflows together.

Prototyping audio products

Explore signal-flow concepts, control schemes, and remote management ideas before narrowing them into a smaller product.

Studying network audio concepts

Use AVB, discovery, and multi-node documentation as a practical path into deterministic audio-over-Ethernet systems.

Headless and remote audio control

Operate a Linux-based audio system from the browser or terminal without depending on a full local desktop workflow.

Amateur recording and practice setups

Turn spare hardware into a customizable audio processor for home labs, practice rooms, and experimentation benches.

Inspecting full-stack engineering tradeoffs

See where a broad platform is strong, where maturity is uneven, and why product closure matters as much as feature count.

Exploring alternative open audio tooling

Use MAP2 as a reference point for what an open, Linux-first musician platform can look like outside commercial black boxes.
